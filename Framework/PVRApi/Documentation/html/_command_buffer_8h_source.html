19 /*!*********************************************************************************************************************
20 \brief Contains all the commands and states that need to be submitted to the gpu, including pipeline, texture, and
21  samplers. Virtually everything that needs to happen on the GPU is submitted to the CommandBuffer.
22  In debug builds (define DEBUG or define PVR_STORE_STACK_TRACE_WITH_API_COMMANDS), a limited stack trace
23  is stored with each command so that if an error occurs, the site where the command was actually added to the
24  command buffer can be determined.
25 
26  Primary command buffers can contain RenderPasses, and can be submitted to the GPU.
27  Secondary command buffers cannot contain RenderPasses, and can be enqueued to PrimaryCommandBuffers.
28 
29  -It is invalid to submit commands to a command buffer while it is not being recorded.

294  /*!****************************************************************************************************************
295  \brief This is the function of choice for updating uniforms (if supported by the underlying API). This function
296  sets a Uniform variable to be updated from a memory location every time the command buffer is submitted, so
297  that updating the value in that memory location is properly updated in the shader uniform.
298  \tparam _type The type of item to upload to the uniform. Supported types depend on the underlying API and shader
299  glsl version. Vectors and matrices should be uploaded using the glm::vec/mat types (not float)
300  \param location The location of the uniform variable (as returned by Pipeline->getUniformLocation)
301  \param count The number of values to read from the pointer
302  \param ptr A pointer to a location that shall remain fixed as long as the command buffer is used (unless cleared).
303  The value will be read from ptr and the uniform updated every time the command buffer is submitted.
304  \description WARNING: This memory location is intended to be fixed, and must remain live as long as the command buffer is
305  used, since every time the command buffer is submitted, this memory location will be read. In synchronous
306  underlying APIs (OGLES) the value can be changed freely without any synchronization before submitting the
307  command buffer or after it returns. Uploading arrays of values is supported.
